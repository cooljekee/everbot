# Skill: Project Setup (Nick Saraev Methodology)

Ты — мастер настройки проекта по методологии курса "Claude Code: Build & Sell 2026".
Когда этот скилл активирован, проведи пользователя через полный setup нового проекта.
Действуй строго пошагово, не переходи к следующему шагу без подтверждения.

---

## STEP 1 — Сбор информации о проекте

Задай пользователю следующие вопросы (можно одним сообщением):

1. **Название проекта** — как будет называться?
2. **Тип проекта** — веб-приложение / API / CLI / агент / другое?
3. **Тех-стек** — язык, фреймворки, БД?
4. **Главная цель** — что должен делать продукт в одном предложении?
5. **Монетизация** — личный проект или для продажи клиентам?

После получения ответов переходи к Step 2.

---

## STEP 2 — Создание CLAUDE.md

Создай файл `CLAUDE.md` в корне проекта. Используй строго этот шаблон (не превышай 200 строк):

```markdown
# [Project Name]

## Overview
[Одно предложение — что делает продукт и для кого]

## Tech Stack
- Language: [язык]
- Framework: [фреймворк]
- Database: [БД]
- Deployment: [куда деплоим]

## File Structure
[Основные директории и их назначение]

## Development Commands
```bash
# Install
[команда установки зависимостей]

# Dev server
[команда запуска]

# Build
[команда сборки]

# Tests
[команда тестов]
```

## Rules & Conventions
- [Соглашения по именованию]
- [Стиль кода]
- [Обязательные проверки перед коммитом]

## Workflow
Research → Plan → Execute → Review → Ship

1. **Research**: изучи кодовую базу перед изменениями
2. **Plan**: используй Plan Mode (`/plan`) для нетривиальных задач
3. **Execute**: реализуй минимально необходимое, без излишних абстракций
4. **Review**: проверь работу перед деплоем — AI не непогрешим
5. **Ship**: деплой только после проверки

## Context Management
- Используй `/compact` когда контекст ~40% заполнен
- Для research-задач запускай субагентов (`/agent`)
- Для параллельной работы используй Git Worktrees
- MCP-серверы увеличивают расход токенов — следи за балансом

## Forbidden Patterns
- Не деплоить без проверки вывода Claude
- Не коммитить .env файлы
- Не игнорировать ошибки типов / линтера
- Не создавать абстракции "на вырост"
```

## STEP 3 — Структура .claude/

Создай следующие директории и файлы:

```
.claude/
├── rules/
│   └── coding-standards.md    ← правила под тип проекта
├── commands/
│   └── review.md              ← команда /review
└── skills/
    └── project-setup/
        └── SKILL.md           ← текущий файл
```

**`.claude/rules/coding-standards.md`** — заполни правилами под тех-стек проекта:
- Соглашения именования переменных/функций
- Обязательное покрытие тестами (если применимо)
- Запрещённые паттерны (any в TS, SELECT * в SQL и т.д.)

**`.claude/commands/review.md`** — шаблон команды `/review`:
```markdown
Проведи code review последних изменений:
1. Корректность логики
2. Безопасность (инъекции, XSS, открытые секреты)
3. Производительность
4. Соответствие coding-standards.md
Дай конкретные замечания с номерами строк.
```

## STEP 5 — Hooks (опционально)

Спроси пользователя: "Нужны ли автоматические hooks?"

Если да — добавь в `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|Bash",
        "hooks": [
          {
            "type": "command",
            "command": "git stash --include-untracked -m 'claude-backup-$(date +%s)' 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
```

> Этот hook делает git stash перед каждым изменением файлов — страховка от потери работы.

## Напоминания по методологии курса

**Главные принципы Nick Saraev:**
- CLAUDE.md — это твой главный актив. Обновляй его по мере роста проекта.
- Всегда верифицируй вывод Claude перед деплоем.
- Hooks > инструкции: hooks детерминированы, инструкции — рекомендательны.
- Один субагент = один фокус. Не делай субагентов-универсалов.
- Git Worktrees — суперсила для параллельной работы над разными ветками.
- Для сайтов: дизайн-спека → компоненты → деплой на Vercel.
- Монетизация: $30K–$50K сделки достигаются через агентные решения под HIPAA/GDPR.

## Управление сессией (новые практики)

**Экстренные команды:**
- `ESC` — мгновенно останавливает Claude если видишь что он пошёл не туда
- `ESC ESC` или `/rewind` — откат к предыдущей рабочей точке
- `/clear` — полный сброс контекста между несвязанными задачами
- `Ctrl+B` — запустить задачу в фоне, продолжая работать

**Правило двух исправлений:**
Если Claude не справился после двух попыток исправить одно и то же — не проси ещё раз. Нажми `/clear` и полностью перепиши промпт с нуля. Накопленный неправильный контекст только мешает.

**Самопроверка через feedback loop:**
Включай в промпт команду для верификации результата:
```
Реализуй X. После реализации запусти `npm test` и убедись что все тесты зелёные.
Если тест упал — исправь и запусти снова.
```
Claude видит вывод тестов, сам исправляет ошибки. Даёт 2–3× рост качества.

**Мониторинг токенов:**
- `/cost` — текущий расход токенов за сессию
- Целевой показатель cache hit: **70%+** на повторяющихся агентах
- Три метрики для контроля: `input_tokens`, `output_tokens`, `cache_read_input_tokens`

## Структура промптов

**Формула: Роль + Задача + Контекст**
```
Ты [роль].
Твоя задача: [конкретное действие].
Контекст: [необходимый фон, файлы, ограничения].
```

**Привязка файлов через @:**
```
@src/auth/login.ts — исправь проверку токена
```
Точные ссылки на файлы вместо описаний — Claude читает именно нужное.

**Приоритет первым:**
Самое важное требование — в начале промпта, не в конце.

**Просить объяснения:**
Если Claude сделал что-то неожиданное — `Объясни логику последнего решения` перед тем как исправлять. Иначе исправишь симптом, не причину.

## Оптимизация токенов (10 правил)

1. **Явное ограничение scope** — "Работай только в файле `src/api.ts`"
2. **Запрет лишнего чтения** — "Не читай другие файлы без явного запроса"
3. **Ограничение формата вывода** — "Только код, без объяснений"
4. **Точные ссылки на файлы** через `@file` вместо описаний
5. **Батчинг изменений** — несколько правок в одном промпте
6. **Инлайн-контекст** — вставляй нужный код прямо в промпт, не проси читать файл
7. **Ограничение инструментов** — "Используй только Edit, не Bash"
8. **Структурированный вывод** — JSON/YAML вместо прозы там где это возможно
9. **Атомарные задачи** — одна задача = один промпт
10. **Отключай неиспользуемые MCP** — каждый активный MCP занимает ~15% контекста

**Модельный стек (выбирай по задаче):**
| Модель | Когда использовать |
|---|---|
| **Haiku** | Простые задачи, форматирование, шаблоны |
| **Sonnet** | Основная разработка, сбалансированный режим |
| **Opus** | Сложный анализ, архитектурные решения |

## Многоуровневая память CLAUDE.md

Используй иерархию файлов памяти:

```
~/.claude/CLAUDE.md          ← персональные правила (для всех проектов)
./CLAUDE.md                  ← правила проекта (коммитится в репо)
./CLAUDE.local.md            ← локальные переопределения (в .gitignore)
.claude/rules/*.md           ← правила по путям (lazy-load)
src/components/CLAUDE.md     ← правила для конкретной директории
```

**Вложенные CLAUDE.md** в поддиректориях загружаются автоматически когда Claude работает в этой папке. Используй для изоляции контекста: разные правила для `src/`, `tests/`, `scripts/`.

**CLAUDE.local.md** — для секретов окружения, личных предпочтений и временных правил. Добавь в `.gitignore`.
